Personal Project : Serverless Mutliplayer Game Platform

Serverless platform where users can create and join QuakeIII games

I made this project to give an overview of how I work. From architecture to implemention, I talk about every aspect of the project, its value and improvments.

I wanted to make the application available online, to let users test it, but also to show that I am confident with the architecture/infrastructure associated. User actions are restricted, costs are known in advance and notifications alert me about users behaviours and problems.

This project is close to a real professional project, in proportion to cost budget, and I would be glad to discuss it with you.

Architecture

Docker images are uploaded into ECR repositories.

At each front-end react commit, the code is deployed into the serverless amplify website.

A user accesses the amplify website using a public custom domain name. He is redirected to a login page, binded with Cognito data.

After successful login, the user can press button which will call HTTP API, and trigger lambda functions. Some APIs can be called only by users belonging to Cognito admin group.

Lambda functions create and manage Cloudformation game-server stacks. A step function is used, to stop automatically game-servers after a timer.

When a user joins a game, the client retreive static files from a S3 bucket, and a WSS connection is made with the ECS game server task.

Each game server are contained in an ECS task. The task is composed of two docker containers : a proxy and a game server.

A dynamodb table stores all game server data (endpoint url, cloudformation stack name, game server status,…)

AWS resources are deployed using Terraform in Dev and Prod aws accounts. A cloudwatch_event deploy the ressources during work hours, in prod account.

About the game

QuakeIII is a first-person shooter game released in 1999 by id Software.

Initially, the C/C++ game has to be installed on the client machine to be played. But, a port to Js has been created so that it can be launched in a browser.

I decided to use this game for this project, because :
• its license allows me to do it,
• game components can be separated into several bricks (game server, web server, content server) and are containerizable
• this game is a pioneer in the history of video games, and is one of the first I played.
